Ce document formalise le data contract de la plateforme D2F avant la mise en place des endpoints API en lecture. Il présente les entités métier, leurs champs, les relations entre elles, les règles de gestion et les endpoints GET proposés pour exposition derrière l'API Gateway. 1. Architecture d'accès

Tous les appels transitent par un point d'entrée unique — l'API Gateway (port 8080) — qui assure la validation JWT, le rate limiting et le circuit breaker avant de router vers les microservices.

6. Règles de gestion

6.1 Validation d'un besoin de formation

Un besoin passe par trois niveaux d'approbation successifs. Un seul refus à n'importe quel niveau stoppe le processus.

ÉtapeActeurChampValeurs possibles
1Chef de DépartementapprouveChefDepnull = en attente · true = approuvé · false = rejeté
2CUPapprouveCUPnull = en attente · true = approuvé · false = rejeté
3Administrateur D2FapprouveAdminnull = en attente · true = approuvé · false = rejeté

6.2 Unicité des savoirs

Un Savoir est rattaché soit à une Competence, soit à une SousCompetence — jamais les deux simultanément (contrainte applicative).

6.3 Unicité du niveau par enseignant

Un enseignant ne peut avoir qu'un seul niveau de maîtrise par savoir. Contrainte unique en base : (enseignantId, savoirId).

6.4 Génération automatique des certificats

Un certificat est créé automatiquement pour chaque enseignant dont l'inscription est APPROVED lorsque la formation passe à l'état ACHEVE. Le champ delivered est mis à true lors de la remise effective.

8. Pagination, tri et recherche

ParamètreTypeDéfautDescription
pageInteger0Numéro de page (0-based)
sizeInteger20Éléments par page (max 100)
sortStringid,descTri : champ,asc ou champ,desc
Exemple : GET /api/v1/formations?page=0&size=20&sort=dateDebut,desc&etatFormation=PLANIFIE

La recherche textuelle (?keyword=) est insensible à la casse et partielle, portant sur les champs nom, code et description selon l'entité.
